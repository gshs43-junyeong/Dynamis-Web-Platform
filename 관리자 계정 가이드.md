# 관리자 계정 가이드

Dynamis Web Platform 관리자(admin) 계정의 권한 범위와 관리자 콘솔 사용법, 그리고 관리자 계정을 승격/회수하는 절차를 안내합니다.

작성 기준: 2026-07-14 `firebase.rules` 최종 수정본 기준

## 1. 관리자 권한은 두 곳에 "따로" 저장됩니다

1. **Firestore `users/{uid}` 문서의 `role` 필드**
   - 값이 `admin`이면 클라이언트 화면에 관리자 메뉴/버튼이 보입니다.
   - 오직 **"화면에 뭘 보여줄지"만** 결정합니다 (UI 표시용).
2. **Firebase Auth 커스텀 클레임 (`admin: true`)**
   - 로그인 토큰(ID Token) 안에 심어지는 값입니다.
   - Firestore 보안 규칙(`firebase.rules`)이 실제로 확인하는 건 **이것**입니다.
   - 즉, "실제로 경고/등급변경/삭제 등의 쓰기 작업이 되는지"를 결정합니다.

> ⚠️ 이 둘은 서로 자동으로 동기화되지 않습니다. `role`만 `admin`으로 바꾸고 커스텀 클레임을 안 심으면, 화면엔 관리자 메뉴가 뜨는데 버튼을 눌러도 "Missing or insufficient permissions" 에러만 계속 납니다.
>
> (원래는 Firestore `get()`으로 `role`을 직접 조회해서 판단하는 방식도 시도했으나, 이 프로젝트 환경에서 원인 불명으로 `get()`이 정상 동작하지 않아 커스텀 클레임 방식으로 전환했습니다. 2026-07-14)

일반 가입은 항상 `role: 'general'`로 생성되며, 본인이 스스로 관리자로 승격할 수 있는 경로는 없습니다. 등급(`role`) 변경은 이미 관리자 클레임을 가진 계정만 콘솔에서 수행할 수 있습니다.

## 2. 새 계정을 관리자로 승격시키는 방법

1. 대상 계정이 이미 Dynamis에 회원가입되어 있어야 합니다 (Google/GitHub 로그인으로 정상 가입 완료 상태).
2. 대상 계정의 uid를 확인합니다.
   - Firebase 콘솔 → Firestore Database → `users` 컬렉션 → 대상 계정의 문서를 열어 `uid` 필드 값을 복사합니다 (문서 ID 자체가 uid와 동일합니다).
3. (선택) 관리자 콘솔에서 대상의 `role`을 "관리자(admin)"로 변경합니다.
   - Dynamis 사이트 → `/admin` → 대상 유저 등급 변경 → "관리자(admin)".
   - 이건 화면 표시(UI)만 바꿉니다. 아래 4번을 반드시 같이 해야 실제로 작동합니다.
4. 커스텀 클레임을 부여합니다 (실제 권한 부여 — 필수).
   - 프로젝트 루트에서 `set-admin-claim.js` 파일을 엽니다.
   - `TARGET_UID` 값을 2번에서 복사한 uid로 교체합니다.
   - 터미널에서 아래 순서로 실행합니다.
     ```bash
     npm install firebase-admin      # 최초 1회만
     node set-admin-claim.js
     ```
   - 아래처럼 출력되면 성공입니다.
     ```
     완료. 현재 customClaims: { admin: true }
     ```
   - `serviceAccountKey.json` 파일이 같은 폴더에 있어야 합니다 (없으면 Firebase 콘솔 → 프로젝트 설정 → 서비스 계정 → "새 비공개 키 생성"으로 발급 후 파일명을 `serviceAccountKey.json`으로 저장).
5. 대상 계정이 직접 **로그아웃 후 재로그인**합니다.
   - 커스텀 클레임은 "새로 발급되는 로그인 토큰"에만 반영됩니다. 새로고침만으로는 절대 반영되지 않습니다.
6. 관리자 콘솔에서 기능(경고 부여 / 등급 변경 / 계정 삭제)이 정상 작동하는지 테스트합니다.

## 3. 관리자 권한을 회수(강등)하는 방법

1. 관리자 콘솔에서 대상 `role`을 `general` 등으로 변경합니다 (UI 반영).
2. `set-admin-claim.js`를 다음과 같이 수정해서 실행합니다 (`admin: true` 대신 `admin: false` 또는 `null`로 설정).
   ```js
   await auth.setCustomUserClaims(TARGET_UID, { admin: false });
   ```
   - 스크립트 안의 이 한 줄만 임시로 바꿔서 실행 후 다시 원상복구하는 것을 권장합니다. 또는 별도의 `remove-admin-claim.js`를 만들어 관리해도 됩니다.
3. 대상 계정 로그아웃 → 재로그인이 필수입니다 (승격 때와 동일한 이유).

## 4. ⚠️ 절대 스스로를 잠그지 마세요 (Lockout 방지)

관리자가 단 1명뿐인 상태에서 그 계정의 admin 클레임을 실수로 `false`로 바꾸면, 이후 누구도 관리자 콘솔에서 admin을 다시 부여할 수 없게 됩니다 (관리자만 등급변경 버튼을 쓸 수 있으므로).

이 경우 복구 방법은 `set-admin-claim.js`를 다시 로컬에서 실행하는 것뿐입니다 (`serviceAccountKey.json`을 가진 사람만 가능). `serviceAccountKey.json`과 `set-admin-claim.js`는 개발/운영 담당자만 접근 가능한 곳에 안전하게 보관해두세요. 최소 1부는 개발자 로컬 또는 안전한 백업 위치에 남겨둘 것.

## 5. 관리자 콘솔 기능

관리자로 로그인하면 상단 네비게이션에 **관리자(Admin)** 메뉴가 나타납니다.

### 등급 변경
회원 목록에서 등급(`general`/`honored`/`member`/`admin`)을 드롭다운으로 선택하고 **변경 확정**을 누르면 즉시 반영됩니다.
- `general`: 등급 없음 (가입 직후 기본값)
- `honored`: 명예부원 (공지/이벤트 작성 가능, 파일 첨부 불가)
- `member`: 부원 (공지/이벤트 작성 및 파일 첨부 가능)
- `admin`: 관리자 (모든 권한 + 관리자 콘솔 접근, 단 2절의 커스텀 클레임 부여가 함께 되어야 실제로 작동)

### 경고 조치
**경고 조치** 버튼을 누르면 해당 계정의 누적 경고 횟수가 1회 증가하고, 다음 로그인 시 경고 알림이 표시됩니다. 경고 누적에 따른 자동 정지/삭제 처리는 없으며, 계정 강제 탈퇴 여부는 관리자가 직접 판단해 별도로 수행합니다.

### 계정 삭제 (강제 탈퇴)
**계정 삭제** 버튼은 대상 계정을 강제 탈퇴시킵니다.
- 대상이 작성한 공지사항/이벤트/댓글을 best-effort로 함께 정리합니다 (일부 정리가 실패해도 계정 삭제 자체는 진행됩니다).
- 대상의 일일 트래픽 통계는 규칙상 본인만 읽고 쓸 수 있어 관리자가 지울 수 없습니다.
- 대상의 Firebase Auth 로그인 계정 자체는 관리자가 삭제할 수 없습니다. Firestore 상의 계정 정보만 삭제되므로, 그 사람이 재로그인을 시도하면 "가입되지 않은 계정"으로 처리되어 재가입이 필요합니다.

### 공지사항 고정
공지사항 목록에서 관리자에게만 보이는 **고정** 체크박스로 특정 공지를 상단에 고정할 수 있습니다.

## 6. 보안 유의사항

1. `serviceAccountKey.json`은 Firebase 프로젝트 전체에 대한 관리자급 비밀 키입니다.
   - 절대 GitHub 등 공개 저장소에 커밋하지 말 것.
   - `.gitignore`에 반드시 포함되어 있어야 함 (이미 설정됨).
   - 다른 사람과 공유 금지, 유출 의심 시 Firebase 콘솔 → 프로젝트 설정 → 서비스 계정에서 즉시 키 폐기 후 재발급.
2. `set-admin-claim.js`는 로컬에서 1회성으로 실행하는 스크립트입니다.
   - 절대 웹사이트 코드(`app.js`, `auth.js` 등)에 포함시키거나 클라이언트에서 실행되게 하면 안 됩니다 (권한 상승 취약점이 됨).
3. 배포 파일명에 주의하세요.
   - 이 프로젝트는 `firebase.json`이 `firestore.rules`가 아니라 `firebase.rules`라는 파일명을 가리키고 있습니다.
   - 두 이름의 파일이 동시에 존재하면 배포가 엉뚱한 파일을 대상으로 이루어져 수정사항이 반영 안 되는 것처럼 보이니, 항상 `firebase.json`의 `firestore` → `rules` 값을 먼저 확인하고 그 경로의 파일만 수정할 것.
4. 규칙 배포 후 콘솔 로그에 `[W]` 경고가 없는지 확인하세요. 경고가 있어도 "Deploy complete!"는 뜨지만, 그 경고가 걸린 함수(예: 이전의 `startsWith`)는 실제로는 오작동할 수 있습니다.
5. 관리자 권한 남용(사적 목적의 정보 열람/유출 등)은 커뮤니티 이용 가이드라인 제4조에 의해 관리자 역할 박탈 및 계정 삭제 대상이 됩니다.
